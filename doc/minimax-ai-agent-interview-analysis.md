## 这套面试在考什么

表面上题很多，实际上围绕 5 个核心能力：

1. 你是否做过真实的 AI 项目，而不是只会调模型 API。
2. 你是否理解 `Agent = 模型 + 记忆 + 工具 + 规划 + 执行 + 评估`。
3. 你是否能把大模型问题落到工程和系统设计层面。
4. 你是否具备扎实的基础能力，比如概率、缓存、并发、Linux、算法。
5. 你是否理解 MiniMax 的技术方向，尤其是长上下文、Agent 原生、规模化 RL、成本效率。

## 高频作答原则

- 不要只讲概念，一定带项目落地细节。
- 不要只说“可以优化”，要说清楚“怎么优化、为什么有效、怎么验证”。
- 回答尽量带指标：如 `P95 延迟`、`QPS`、`成功率`、`召回率`、`token 成本`、`幻觉率`。
- 系统设计题一定补上：限流、超时、重试、幂等、日志、监控、降级。
- RAG 和 Agent 相关题不要把答案说成“把所有上下文都塞给模型”。

### 2. 介绍简历上最相关的 AI 项目，重点讲技术选型、难点和解决过程

原题：项目介绍，介绍简历上最相关的 AI 项目，重点讲技术选型、难点和解决过程。

考察点：

- 项目真实性
- 你是否真正主导过关键模块
- 是否具备完整闭环能力

推荐结构：

1. 业务目标
2. 系统架构
3. 关键选型和原因
4. 最大难点
5. 你的解决方案
6. 最终效果

参考回答框架：

> 这个项目的目标是解决什么问题，使用场景是什么，为什么传统规则或普通搜索不够。整体架构上，我把系统拆成了数据接入、索引构建、召回与重排、上下文组装、模型推理、工具调用和评测监控几层。选型上，比如为什么用了向量检索加关键词混合召回，为什么需要 rerank，为什么对会话做摘要而不是保存全量历史。最大的难点通常是召回准确率不稳定、长尾问题覆盖差、工具调用结果不可靠，或者高峰期延迟抖动。解决上可以讲你做的 query rewrite、chunk 优化、缓存策略、并行调用、失败重试、trace 监控。最后一定量化结果，比如任务成功率提升、平均响应时间下降、人工兜底比例下降。

面试里最好能直接说出的两句：

- “我不只负责提示词，而是负责从检索、上下文构建到服务上线稳定性的整条链路。”
- “我会优先用离线评测集和线上指标联合判断方案，而不是只看单次 demo 效果。”

### 3. 解释 Transformer 的注意力机制，以及 MiniMax Lightning Attention 相比传统注意力的优势

考察点：

- 模型基础是否扎实
- 是否理解长上下文技术路线
- 是否了解 MiniMax 的公开技术方向

参考回答：

Transformer 的自注意力本质上是让每个 token 根据当前输入序列里其他 token 的相关性，动态决定“该关注谁”。常见实现是先把输入映射成 `Q`、`K`、`V`，再通过 `QK^T` 计算相似度，经过 `softmax` 得到权重，最后对 `V` 做加权求和。它的优点是建模全局依赖能力强，但标准注意力的计算和显存开销会随着序列长度快速增长，长上下文场景下成本很高。

结合 MiniMax 公开资料可以这样答：Lightning Attention 的核心价值不是一句“更快”就结束，而是更适合超长上下文建模。相比标准注意力，它更关注长序列下的吞吐、显存占用和可扩展性，因此在百万级上下文、长文档理解、代码库分析这类任务中更有优势。面试里不一定需要把具体公式全部展开，但一定要把重点落在三个词上：

- 长序列成本更低
- 推理吞吐更高
- 更适合扩展到超长上下文任务

加分表达：

> 我理解 MiniMax 的差异化不只是“上下文更长”，而是“在长上下文下还能保持比较可用的成本和效率”，这对 Agent 尤其重要，因为 Agent 很多任务天然需要长期状态和大量上下文。

### 4. AI Agent 通常包含哪些核心组件？请简述 ReAct 框架的工作流程

考察点：

- Agent 架构理解是否成体系
- 是否知道“Reasoning + Acting”闭环

参考回答：

一个完整的 AI Agent 通常至少包括以下组件：

- `Planner / Reasoner`：决定下一步做什么
- `Memory`：保存短期上下文、长期记忆、用户画像、历史任务状态
- `Tool Use`：调用外部 API、数据库、搜索、执行器
- `Retriever`：检索外部知识或历史信息
- `Executor`：执行工具调用和流程编排
- `Observation`：读取外部环境反馈
- `Evaluator / Guardrail`：评估结果质量、做安全约束和错误恢复

ReAct 的核心是把“思考”和“行动”交替进行。典型流程是：

1. 模型先根据目标做一次推理，决定是否需要行动
2. 如果需要，选择一个工具并生成参数
3. 外部系统执行工具，返回 observation
4. 模型读取 observation，再进行下一轮 reasoning
5. 循环直到得到最终答案或达到终止条件

加分点：

- 说明 ReAct 的优点是可解释、可调试
- 说明缺点是步骤多、成本高、容易级联失败

### 5. 在智能体开发中，如何设计一个可靠的 Function Calling 机制来避免模型幻觉

考察点：

- 工具调用工程经验
- 是否懂“模型不能直接信”

参考回答：

可靠的 Function Calling 不能只依赖模型自己“猜”参数，而需要工程约束。我的做法通常有 6 层：

1. 严格的工具 schema：参数类型、枚举值、必填项必须清晰定义。
2. 参数校验：模型给出的参数先过 JSON Schema 或服务端校验器。
3. 工具白名单：模型只能调用被显式开放的工具，不能随意生成新工具名。
4. 结果回填机制：工具结果必须由系统写回上下文，禁止模型伪造结果。
5. 失败重试和降级：参数缺失、调用超时、结果为空时要有回退策略。
6. 全链路日志与审计：保留 prompt、参数、结果、耗时、错误码，方便定位问题。

一句话版本：

> 核心原则是“模型负责提议，系统负责校验和执行”，不能让模型既当决策者又当最终事实来源。

### 6. 如果让你设计一个支持多轮对话的智能体服务，如何保证长上下文（如 100K+ tokens）的稳定性和效率

考察点：

- 长上下文架构设计
- 记忆管理是否合理

参考回答：

我不会把所有历史对话无脑拼进去，而是做分层记忆：

- 短期记忆：最近几轮对话，保留高分辨率信息
- 中期记忆：对较早历史做结构化摘要
- 长期记忆：用户画像、偏好、事实信息，独立存储
- 外部检索：对知识类内容通过向量检索或关键词检索按需取回
- 任务状态：当前 Agent 已经执行到哪一步，单独保留

为了保证稳定性和效率，我会加这些机制：

- 上下文裁剪和去重
- 会话摘要滚动更新
- 前缀缓存和 KV Cache
- 热点问答缓存
- 工具结果结构化存储，避免重复生成
- 对长文档先检索后阅读，而不是全量塞入

面试里的高分句：

> 长上下文不是“存更多文本”，而是“让真正有用的信息在当前回合以合适粒度出现”。

### 7. 手写一个简单的 RAG 相似度匹配算法

原题：实现一个简单的 RAG 流程中的相似度匹配算法，如 BM25 或向量余弦相似度计算。

考察点：

- 检索基础是否扎实
- 是否能写出核心代码

答题思路：

- 如果题目偏手写，优先实现余弦相似度，简单直接
- 如果题目偏原理，补充 BM25 的词频、逆文档频率、长度归一化思想

可直接口述的说明：

> 向量检索适合做语义匹配，BM25 更适合做关键词精确匹配。实际线上系统里我通常会做混合召回，因为只用向量容易语义漂移，只用 BM25 又容易漏召回。

### 8. LeetCode 470：用 Rand7() 实现 Rand10()

考察点：

- 概率基础
- 均匀分布理解
- 拒绝采样优化

参考思路：

两次调用 `rand7()` 可以构造出均匀分布的 `1..49`。只取 `1..40`，将其映射到 `1..10`，如果落在 `41..49` 就重试。这样每个结果出现概率一致。

高分点：

- 能解释为什么这样是均匀的
- 知道可继续回收 `41..49` 的残余区间做进一步优化

### 9. Linux 下如何批量处理大量文件，并考虑并行化

考察点：

- 基础工程能力
- 是否有批处理经验

参考回答：

常见做法是用 `find` 找文件，用 `xargs -P` 或 `GNU parallel` 做并行，再结合 `awk`、`sed`、`grep`、自定义脚本完成重命名、内容提取或转换。真正在线上做时，我会重点关注：

- 幂等性：同一批任务重复执行不会出错
- 日志与失败重试
- 并行度控制，避免把磁盘或 CPU 打满
- 分批处理和进度记录
- 先在样本集验证再全量执行

### 2. 项目深挖：在 RAG 系统中，如何解决“长尾问题”或“数据稀疏性”

考察点：

- 你是否做过真实检索优化
- 是否知道冷启动和长尾场景的常见解决办法

参考回答：

长尾问题通常是查询表达多样、知识覆盖不足、召回逻辑泛化差导致的。我的处理思路一般分三层：

1. Query 层：做 query rewrite、同义词扩展、意图分类。
2. Recall 层：混合召回，加入关键词检索、向量检索和规则召回。
3. Rank 层：使用 rerank、业务特征、来源可信度做精排。

对于数据稀疏性，尤其是新领域或冷启动场景，我会做：

- 结构化知识补充
- FAQ 和规则兜底
- 从日志中构建 hard case 数据集
- 利用用户反馈反哺索引和评测集

高分句：

> 长尾问题本质上不是单点模型能力不足，而是查询理解、知识覆盖和排序策略共同作用的结果。

### 3. 了解 MiniMax 自研 Forge 框架吗？请谈谈你对 Agent 原生模型（如 M2.5）的理解，以及它在成本控制上的优势

考察点：

- 是否认真研究过公司路线
- 是否能理解 Agent 原生和通用聊天模型的区别

参考回答：

基于 MiniMax 公开资料，Forge 更像是一个面向 Agent 训练和扩展的底层框架，它把训练、推理、数据和 agent scaffold 做了解耦，目标是支持不同类型的 Agent 任务进行大规模强化学习和能力迭代。它不是简单的应用层 SDK，而更偏底层平台能力。

我理解“Agent 原生模型”的意思，是模型不是仅针对问答或聊天优化，而是在训练目标和能力结构上，更强调：

- 工具调用
- 长流程任务分解
- 环境交互
- 多步执行与自我修正
- 面向真实任务完成而不是只生成文本

至于成本优势，我会从三个角度回答：

1. 长上下文下的单位计算成本更优
2. 对工具调用和 Agent 工作流更友好，减少无效推理轮次
3. 如果训练和推理系统是围绕 Agent 场景设计的，整体资源利用率通常更高

稳妥表达：

> 我的理解是，MiniMax 的重点不是只做“更大的聊天模型”，而是做更适合复杂任务执行的 Agent 原生能力，并且尽量把这种能力建立在可控成本之上。

### 4. 在智能体训练中，PPO 和 DPO 有什么区别？为什么 DPO 在微调中更受欢迎

考察点：

- 对齐训练方法是否懂基本原理
- 是否知道工业界为什么常用 DPO

参考回答：

PPO 属于强化学习方法，通常需要 reward model、在线采样、策略更新，训练链路复杂，但适合直接优化复杂行为目标。DPO 则是直接利用偏好数据对模型进行优化，不需要单独训练 reward model，也不需要完整 RL 环节，工程实现更简单，训练更稳定。

为什么 DPO 更受欢迎：

- 数据准备相对明确，偏好对容易构造
- 训练链路更简单，成本更低
- 相比 PPO 更稳定，调参难度更低
- 对很多“回答质量偏好”类问题已经足够有效

同时也要补一句：

> DPO 适合偏好对齐，但如果目标是复杂长流程决策，尤其是 Agent 在环境中的长期奖励优化，RL 仍然有价值。

### 5. 解释 LoRA 的原理，以及它在智能体微调中的应用场景

考察点：

- 微调方法基础
- 是否知道 LoRA 的工程价值

参考回答：

LoRA 的核心思想是冻结原始大模型参数，只在某些线性层上引入低秩可训练矩阵，通过学习一个较小的增量来完成适配。这样做的好处是训练参数少、显存占用低、部署灵活。

在 Agent 场景里，LoRA 比较适合：

- 对工具调用格式进行对齐
- 针对某个垂类强化专业术语和任务风格
- 提升特定工作流的指令遵循能力
- 做多租户或多场景快速适配

不太适合的情况：

- 想靠 LoRA 单独解决复杂推理能力不足
- 数据量太小且质量很差

### 6. 设计一个支持高并发请求的智能体服务架构，如何做负载均衡和容错处理

考察点：

- 后端系统设计能力
- 是否有生产环境意识

参考回答：

一个常见架构可以这样设计：

- 接入层：API Gateway，做鉴权、限流、路由
- 编排层：任务调度器，管理对话状态、工具链和工作流
- 推理层：模型服务，支持多实例扩容
- 工具层：搜索、数据库、第三方 API、代码执行器等
- 存储层：缓存、会话状态库、日志库、向量库
- 监控层：指标、日志、trace、告警

负载均衡上：

- 网关层按租户、模型、流量做路由
- 推理层支持多实例横向扩容
- 工具服务按类型拆分，避免互相影响

容错上：

- 限流、熔断、超时控制
- 重试但要区分幂等和非幂等操作
- 结果缓存和降级兜底
- 任务级 trace，便于定位哪一步失败

高分点：

- 明确区分“模型推理失败”和“工具执行失败”
- 说明要把推理层和工具层分开扩缩容

### 7. 当智能体处理超长文本（如整个代码库）时，除了分块，还有哪些技术手段可以提升推理速度

考察点：

- 性能优化思路是否丰富

参考回答：

除了简单分块，我通常还会考虑：

- 检索裁剪：先找最相关片段，再读全文
- 层级摘要：先模块摘要，再文件摘要，再全局汇总
- Prefix Cache / KV Cache：复用相同前缀的计算
- 并行子任务：把任务拆成独立子问题并发处理
- 工具前置：先做静态分析、依赖分析，再交给模型
- 异步 I/O 和结果复用：避免重复读取和重复推理
- 模型路由：简单问题走小模型，复杂问题走大模型

一句话版本：

> 真正提升速度，关键不只是“切得更细”，而是“让模型只读必要信息，并尽量复用已经算过的东西”。

### 8. 如何评估一个 AI Agent 的好坏？除了准确率，还会关注哪些指标

考察点：

- 是否具备评测思维
- 是否懂线上指标

参考回答：

我通常把 Agent 评估拆成 4 类：

1. 效果指标：任务成功率、答案正确率、工具调用成功率、幻觉率
2. 效率指标：平均步骤数、平均响应时间、P95 延迟、token 消耗
3. 成本指标：单请求成本、GPU 利用率、缓存命中率
4. 稳定性指标：错误率、超时率、人工接管率、回退率

如果是业务系统，我还会加：

- 用户满意度
- 留存或转化
- 每个任务节省的人力时间

高分句：

> Agent 不能只看最后答案对不对，还要看它为了得到答案付出了多少步、多少成本，以及过程中有没有稳定、可控、可复现。

### 9. 手写代码，实现一个简单的 KV 缓存机制，支持 TTL 过期和 LRU 淘汰

考察点：

- 数据结构能力
- 代码实现细节

参考回答：

标准做法是 `HashMap + 双向链表`：

- `HashMap`：O(1) 找到 key 对应节点
- 双向链表：维护最近使用顺序，便于 LRU 淘汰
- 节点附带 `expireAt` 字段支持 TTL

实现时要考虑：

- `get` 时如果已过期，要删除并返回空
- `put` 时如果存在旧值，要更新并移动到链表头
- 超容量时淘汰链表尾部
- TTL 清理可以懒删除，也可以后台定期清理
- 并发场景要考虑锁或分段锁

## 三面

### 1. 对比当前主流的智能体开发框架（如 LangChain vs CrewAI vs MiniMax Forge），你更倾向于哪种？为什么

考察点：

- 技术视野
- 是否能做框架层面的判断

参考回答：

我不会简单说哪个“最好”，而是按场景选。

- `LangChain`：生态成熟、组件多，适合快速搭建单 Agent 或工作流原型。
- `CrewAI`：更强调多 Agent 协作和角色分工，适合做协同式任务编排。
- `MiniMax Forge`：从官方资料看，更偏底层 Agent 训练和规模化迭代平台，而不是只做应用层封装。

如果是快速验证业务，我可能先选更成熟的应用框架；如果目标是构建长期演进的 Agent 平台，尤其关注训练、评测和大规模迭代能力，我会更关注像 Forge 这种更底层的系统。

高分句：

> 我更在意框架是否匹配目标阶段：原型期看开发效率，平台期看可观测性、评测、扩展性和成本。

### 2. 你认为 AI Agent 在哪些垂直领域最容易实现商业化落地？为什么

考察点：

- 业务理解
- 是否具备 ROI 思维

参考回答：

我认为最容易落地的几个方向是：

- 编程助手：输入输出明确，ROI 可量化，用户愿意付费
- 办公自动化：文档处理、报告生成、流程编排，替代重复劳动
- 企业知识问答：结合内部文档，能显著降低检索和支持成本
- 客服和销售辅助：标准流程多，效果易度量

原因主要有三点：

1. 任务边界相对清晰
2. 成果可以量化，比如节省时间、提升转化
3. 业务流程本身已数字化，适合 Agent 接入

### 3. 最近半年你关注了哪些 AI 领域前沿技术或论文？请分享一篇印象深刻的论文

考察点：

- 学习能力
- 是否真的跟踪前沿

推荐答法：

- 不要一次讲太多
- 选 1 篇你真的能讲清楚的论文
- 按“问题、方法、贡献、局限、启发”来讲

参考模板：

> 我最近比较关注长上下文、Agent RL、推理增强和代码智能体这几个方向。印象比较深的一篇是 MiniMax-01 相关工作，因为它体现了长上下文架构设计和成本控制的结合。我的关注点不是只看 benchmark，而是看这种架构是否真的适合真实 Agent 任务，比如长文档、多轮状态和工具协作。在项目上它给我的启发是，长上下文能力不是为了展示窗口大小，而是为了让系统能更稳定地处理跨步骤、跨文档的信息。

### 4. 在开发过程中，如果团队成员对技术方案有分歧，例如是否使用 RAG vs 全量上下文，你会如何处理

考察点：

- 协作能力
- 是否能用数据驱动决策

参考回答：

我一般不会先讨论“谁对谁错”，而是先统一目标，比如要优化的是准确率、成本、还是延迟。然后把争议转化成可以验证的实验：

1. 明确评测集和验收指标
2. 做 A/B 或离线实验
3. 比较效果、延迟、成本、稳定性
4. 复盘方案适用边界

如果是 `RAG vs 全量上下文`，我通常会这样看：

- 知识量大、更新快、需要可追溯，优先 RAG
- 上下文规模可控、任务依赖强连续性，可能考虑更长上下文
- 实际上很多系统是混合方案，而不是二选一

高分句：

> 我会尽量把“观点分歧”变成“实验设计问题”，最终用数据而不是资历来做技术决策。

### 5. 描述一次你在项目中遇到的最大技术挑战，以及你是如何解决的

考察点：

- 抗压能力
- 问题分析和闭环能力

推荐结构：STAR

- `Situation`：背景
- `Task`：你要解决什么
- `Action`：你做了什么
- `Result`：结果如何

适合 AI 项目的挑战主题：

- RAG 命中率低导致线上答非所问
- 工具调用链路不稳定
- 上下文过长导致成本过高和延迟过大
- 高并发下模型服务不稳定

回答时要突出：

- 你怎么定位问题
- 你为什么做出那个方案选择
- 你如何验证真的解决了问题

### 6. 你未来 3 到 5 年的职业规划是什么？为什么选择 AI Agent 这个方向

考察点：

- 动机是否真实
- 是否愿意长期投入

参考回答：

> 我希望未来 3 到 5 年，先把自己打磨成能独立负责 AI Agent 复杂项目的人，不只是会接模型，还能把检索、工具、评测、服务架构和业务落地串起来。中期我更希望往 Agent 平台或智能体系统方向发展，参与更底层的能力建设，比如评测体系、工作流框架、训练和推理协同。之所以选择 AI Agent，是因为它比传统聊天系统更接近真实生产力，我对“让模型真的完成任务”这件事更有兴趣。

面试官想确认你不是“提示词工程师”，而是能真正把系统搭起来的人。

- 降级
- 监控
