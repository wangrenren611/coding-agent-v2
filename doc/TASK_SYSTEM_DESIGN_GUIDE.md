## 1. 系统概述

### 1.1 为什么有两套系统？

在实际的软件开发任务中，存在两个层面的问题：

**第一个层面：做什么（规划层面）**
- 一个大任务需要拆分成多个小任务
- 小任务之间有先后顺序（依赖关系）
- 需要追踪每个任务的进度
- 需要知道哪些任务可以开始，哪些需要等待

**第二个层面：怎么做（执行层面）**
- 实际执行代码探索、命令运行、网络搜索等工作
- 某些工作需要特殊的技能（比如专门的代码搜索）
- 某些工作可以并行执行
- 某些工作耗时较长，需要在后台运行

**解决方案**：使用两套独立的系统分别处理这两个层面

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   TaskCreate/Update/Get/List ────▶ 管理做什么（任务列表）   │
│                                                             │
│   Task ──────────────────────────▶ 管理怎么做（实际执行）   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 系统角色类比

想象一个软件开发团队：

| 现实团队角色 | 对应工具系统 | 职责 |
|-------------|-------------|------|
| 项目经理 | TaskCreate/Update/Get/List | 制定计划、分配任务、追踪进度 |
| 开发工程师 | Task (子代理) | 写代码、调试、测试 |
| 测试工程师 | Task (Bash代理) | 运行测试、报告结果 |
| 架构师 | Task (Plan代理) | 设计方案、技术调研 |
| 研究员 | Task (Research代理) | 搜索资料、获取信息 |

## 2. 核心概念解释

### 2.1 任务（Task）的概念

在 TaskCreate 系统中，一个"任务"包含以下信息：

**基本信息**
- **任务ID**：唯一标识符，比如 "1"、"2"、"3"
- **标题**：简短描述要做什么，比如"实现用户登录功能"
- **详细描述**：具体的验收标准和要求
- **进行中状态描述**：任务执行时显示的文本，比如"正在实现用户登录功能"

**状态信息**
- **待处理（pending）**：任务已创建，等待开始
- **进行中（in_progress）**：任务已被认领，正在执行
- **已完成（completed）**：任务已完成

**关系信息**
- **被谁阻塞（blockedBy）**：必须先完成哪些任务才能开始这个任务
- **阻塞了谁（blocks）**：这个任务完成后，哪些任务才能开始

**归属信息**
- **所有者（owner）**：哪个代理或用户正在处理这个任务

### 2.2 依赖关系详解

**什么是依赖关系？**

假设你要"刷墙"，必须先"买油漆"。那么：
- "刷墙" 依赖于 "买油漆"
- "买油漆" 阻塞了 "刷墙"
- 只有 "买油漆" 完成后，"刷墙" 才能开始

**依赖关系的方向**

```
买油漆  ──────▶  刷墙

箭头方向表示：前者完成后，后者才能开始
```

**多级依赖**

```
赚钱  ──▶  买油漆  ──▶  刷墙  ──▶  晾干  ──▶  贴画

每个任务都必须等待前面的任务完成后才能开始
```

**并行任务**

```
                    ┌──▶  买油漆
赚钱  ──▶  去超市 ──┤
                    └──▶  买刷子

"买油漆" 和 "买刷子" 可以同时进行，它们都只依赖 "去超市"
```

### 2.3 子代理（Subagent）的概念

**什么是子代理？**

子代理是一个独立运行的"助手"，它有自己的：
- 专长领域（比如擅长执行命令、擅长搜索代码）
- 可用工具（有些代理只能用某些工具）
- 独立上下文（有自己的对话历史）

**子代理类型**

| 代理类型 | 专长 | 典型使用场景 |
|---------|------|-------------|
| Bash | 执行命令 | git操作、npm命令、运行测试 |
| Explore | 搜索代码 | 查找文件、搜索代码内容、理解代码结构 |
| Plan | 设计方案 | 功能设计、架构规划、实现方案 |
| Research | 网络研究 | 搜索最新资料、查阅文档、获取信息 |
| General-purpose | 通用任务 | 复杂多步骤任务、需要多种能力的任务 |

**子代理的执行模式**

- **同步模式**：主程序等待子代理完成后继续
- **后台模式**：子代理在后台运行，主程序可以继续做其他事

## 3. Task 子代理系统详解

### 3.1 Task 工具的执行流程

**步骤一：接收请求**

当调用 Task 工具时，需要提供：
- 选择哪个类型的代理
- 告诉代理要做什么（任务描述）
- 可选的配置（模型选择、后台运行等）

**步骤二：创建代理实例**

系统会：
1. 生成一个唯一的代理ID
2. 根据代理类型获取对应的配置（可用工具、默认模型等）
3. 初始化代理的上下文（对话历史）
4. 构建系统提示词（告诉代理它的角色和任务）

**步骤三：判断执行模式**

系统判断是同步执行还是后台执行：

- **如果选择后台运行**：
  1. 创建一个输出文件用于记录执行日志
  2. 启动后台进程
  3. 立即返回代理ID和输出文件路径
  4. 主程序继续执行其他任务
  5. 后续可以通过 TaskOutput 工具获取结果

- **如果选择同步运行**：
  1. 进入代理执行循环
  2. 等待代理完成任务
  3. 返回最终结果

**步骤四：代理执行循环（同步模式下）**

代理进入一个循环，每次循环包括：

1. **思考阶段**：代理分析当前情况，决定下一步行动
2. **行动阶段**：
   - 如果代理认为任务完成，退出循环
   - 如果代理需要使用工具，执行工具调用
   - 如果代理需要更多信息，请求用户输入
3. **更新上下文**：将本次交互添加到对话历史
4. **检查限制**：是否超过最大轮次限制

**步骤五：返回结果**

执行结束后返回：
- 代理ID（可用于恢复继续执行）
- 执行状态（完成/失败/暂停）
- 输出结果或错误信息

### 3.2 代理如何决定任务完成

代理会在以下情况下认为任务完成：

1. **没有更多工具调用请求**：代理认为不需要再执行任何操作
2. **明确表示完成**：代理在回复中说明任务已完成
3. **达到外部限制**：超过最大轮次限制，被强制暂停

### 3.3 代理的工具权限

不同类型的代理有不同的工具权限：

**Bash 代理**
- 只能使用 Bash 工具
- 适合执行预定义的命令序列
- 安全性较高，能力受限

**Explore 代理**
- 可以使用 Glob（文件搜索）
- 可以使用 Grep（内容搜索）
- 可以使用 Read（读取文件）
- 可以使用 Bash（简单命令）
- 专注于代码探索，不会修改代码

**Plan 代理**
- 可以使用 Glob、Grep、Read（理解现有代码）
- 可以使用 WebSearch、WebFetch（查阅资料）
- 不执行实际代码修改
- 专注于设计和规划

**Research 代理**
- 可以使用 WebSearch（网络搜索）
- 可以使用 WebFetch（获取网页内容）
- 可以使用 Read、Grep（查阅本地文档）
- 专注于信息收集

**General-purpose 代理**
- 可以使用所有工具
- 能力最强，灵活性最高
- 适合复杂任务

### 3.4 恢复中断的代理

如果代理因为超时或其他原因中断，可以恢复继续执行：

1. 记录中断时的代理ID
2. 使用 Task 工具时传入 resume 参数
3. 系统会恢复之前的对话上下文
4. 代理从中断点继续工作

## 4. TaskCreate 系列工具详解

### 4.1 TaskCreate：创建任务

**执行流程**

**步骤一：参数验证**

系统检查：
- 标题是否存在且足够长（至少3个字符）
- 描述是否存在且足够详细（至少10个字符）
- 进行中描述是否提供（可选，有默认值）

**步骤二：重复检查**

系统遍历现有任务，检查是否有相同标题的未完成任务。如果存在，拒绝创建以避免重复。

**步骤三：创建任务实体**

系统生成：
- 唯一的任务ID（通常是递增的数字）
- 初始化状态为"待处理"
- 清空依赖关系列表
- 记录创建时间

**步骤四：存储任务**

将任务保存到任务列表存储中，并在依赖图中创建对应的节点。

**何时使用**

- 收到用户的多步骤请求时
- 需要拆分复杂任务时
- 需要追踪多个相关工作项时

### 4.2 TaskUpdate：更新任务

**执行流程**

**步骤一：获取最新状态**

系统首先读取任务的当前状态，确保操作基于最新数据。这是为了避免并发修改导致的数据不一致。

**步骤二：状态转换验证**

如果请求修改状态，系统检查转换是否合法：

| 当前状态 | 可以转换到 |
|---------|-----------|
| pending | in_progress |
| in_progress | completed, pending |
| completed | （不能转换） |

**关键判断**：
- pending 只能变成 in_progress（不能直接跳到 completed）
- in_progress 可以变成 completed（成功完成）或 pending（放弃/回退）
- completed 是终态，不能再改变

**步骤三：处理依赖关系**

如果请求添加依赖（addBlockedBy）：

1. **验证任务存在**：检查阻塞任务和被阻塞任务都存在
2. **检测循环依赖**：确保添加依赖后不会形成环
3. **更新依赖图**：在图中添加对应的边
4. **更新任务字段**：更新 blockedBy 和 blocks 列表

**步骤四：更新其他字段**

更新标题、描述、所有者、元数据等字段。

**步骤五：完成通知**

如果状态变为 completed，系统会检查哪些任务依赖于此任务，通知它们可能已经解锁。

**何时使用**

- 开始执行任务时（更新状态为 in_progress，设置 owner）
- 任务完成时（更新状态为 completed）
- 发现新依赖关系时（添加 addBlockedBy）
- 任务失败需要重新分配时（清空 owner，回退状态）

### 4.3 TaskGet：获取任务详情

**执行流程**

**步骤一：查找任务**

根据任务ID在存储中查找任务。

**步骤二：计算依赖状态**

对于每个阻塞此任务的任务，系统检查其状态，生成一个摘要列表，包括：
- 阻塞任务的ID
- 阻塞任务的标题
- 阻塞任务的当前状态

**步骤三：计算被阻塞任务**

找出此任务阻塞的所有任务，生成摘要列表。

**步骤四：计算是否可以开始**

系统执行以下检查：

1. **状态检查**：任务必须是 pending 状态
2. **依赖检查**：所有阻塞任务必须已完成
3. **归属检查**：任务不能已被其他人认领

返回详细的判断结果和原因。

**何时使用**

- 在开始执行任务前，检查是否可以开始
- 查看任务的完整信息
- 了解任务在依赖链中的位置

### 4.4 TaskList：列出所有任务

**执行流程**

**步骤一：遍历所有任务**

系统遍历存储中的所有任务。

**步骤二：计算每个任务的状态**

对于每个任务，计算：
- 是否被阻塞（blockedBy 中是否有未完成的任务）
- 是否可以被认领（pending + 无阻塞 + 无所有者）

**步骤三：排序**

按照以下优先级排序（考虑任务优先级和状态）：
1. 紧急(critical)且可开始的任务 - 最高优先级
2. 进行中的任务
3. 高优先级(high)可认领的任务
4. 普通优先级(normal)可认领的任务
5. 低优先级(low)可认领的任务
6. 被阻塞的任务
7. 已完成的任务(最后显示)
8. 已取消/已失败的任务

**何时使用**

- 查看整体进度
- 寻找下一个可以执行的任务
- 检查是否有任务被阻塞

## 5. 两套系统的协作关系

### 5.1 协作的本质

**TaskCreate 系统的作用**
- 管理"要做什么"
- 维护任务之间的依赖关系
- 追踪整体进度
- 防止遗漏或重复

**Task 系统的作用**
- 执行"如何做"
- 实际读取文件、运行命令、搜索网络
- 返回执行结果

**协作方式**

1. 主代理使用 TaskCreate 创建任务列表
2. 主代理使用 TaskList 查找可执行的任务
3. 主代理使用 TaskGet 确认任务可以开始
4. 主代理使用 TaskUpdate 认领任务
5. 主代理使用 Task 启动子代理执行实际工作
6. 子代理完成后，主代理使用 TaskUpdate 标记任务完成
7. 重复步骤 2-6 直到所有任务完成

### 5.2 完整执行流程图

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     任务系统完整执行流程                                      │
└─────────────────────────────────────────────────────────────────────────┘

                                用户请求
                                   │
                                   ▼
                    ┌───────────────────────┐
                    │  需要任务列表？  │
                    │                 │
              是       │ 否       │ 否
              ▼       ▼        ▼
        ┌───────┴────────┐─────────────┴──────────────┐─────────────┐
        │         │            │             │              │             │
        ▼         ▼            ▼             ▼              ▼             │
  TaskCreate        TaskList      TaskGet        TaskUpdate      Task        TaskOutput
   创建任务          列出任务      获取详情      更新任务      子代理执行    获取结果
        │             │             │             │              │             │
        │             │             │             │              │             │
        └─────────────┴─────────────┴──────────────┴─────────────┴─────────────┘
              │             │             │             │              │
              │             │             │             │              │
              ▼             ▼             ▼             ▼              ▼
        ┌───────────────────────────────────────────────────────────────────┐
        │                     详细执行步骤                                          │
        └───────────────────────────────────────────────────────────────────┘

          1. TaskCreate（创建任务）
             ├── subject: "实现登录功能"
             ├── description: "实现用户登录API..."
             ├── priority: "high"
             └── 返回: taskId

          2. TaskUpdate（设置依赖）
             ├── taskId: "2"
             └── addBlockedBy: ["1"]  // 任务2依赖任务1

          3. TaskList（查看任务）
             └── 返回: [任务1: pending,可认领]
                           [任务2: pending, 被阻塞]

          4. TaskGet（检查任务1）
             └── canStart: { canStart: true }

          5. TaskUpdate（认领任务）
             ├── taskId: "1"
             ├── status: "in_progress"
             └── owner: "main-agent"

          6. Task（启动子代理）
             ├── subagent_type: "general-purpose"
             ├── prompt: "实现登录功能..."
             └── run_in_background: false

          7. TaskUpdate（标记完成）
             ├── taskId: "1"
             └── status: "completed"

          8. TaskList（查看解锁）
             └── 任务2 现在可以认领

          9. 重复步骤 4-7 直到所有任务完成
```

### 5.3 任务分配策略

**如何选择子代理类型？**

根据任务的性质选择：

| 任务特征 | 推荐代理类型 |
|---------|-------------|
| 需要执行 git、npm、docker 命令 | Bash |
| 需要查找文件或搜索代码 | Explore |
| 需要设计方案或架构 | Plan |
| 需要搜索网络获取最新信息 | Research |
| 需要写代码、修改文件 | General-purpose |
| 复杂任务涉及多种操作 | General-purpose |

**何时使用后台运行？**

- 任务耗时较长（超过几分钟）
- 需要同时执行多个任务
- 主程序需要继续处理用户交互

### 5.3 并行执行策略

**识别可并行的任务**

通过依赖图分析：
- 没有依赖关系的任务可以并行
- 依赖同一个任务但彼此独立的任务可以并行

**示例**

```
任务依赖图：

     [任务A]
        │
        ▼      ▼      ▼
     [任务B] [任务C] [任务D]
        │      │      │
        └──────┴──────┘
               │
            [任务E]

并行执行策略：
- 第一波：执行任务A
- 第二波：同时执行任务B、C、D
- 第三波：执行任务E
```

**实现方式**

1. 使用 TaskList 获取所有任务
2. 筛选出 pending 且无阻塞的任务
3. 为每个任务启动一个后台子代理
4. 定期检查完成情况
5. 当有新任务解锁时，重复步骤 2-4

## 6. 完整执行示例：开发一个天气查询功能

### 6.1 用户请求

用户说："帮我实现一个天气查询功能，用户可以输入城市名，返回该城市的天气信息。需要选择合适的天气API，实现前端界面和后端接口，并编写测试。"

### 6.2 第一阶段：理解需求与任务拆分

**主代理的思考过程**

1. 这是一个复杂的多步骤任务
2. 需要拆分成多个子任务
3. 子任务之间有依赖关系
4. 适合使用 TaskCreate 系统管理

**创建任务列表**

主代理决定创建以下任务：

**任务1：调研天气API**
- 标题：调研天气API
- 描述：搜索并比较不同的天气API服务，选择最适合的方案，记录API的使用方法、定价、限制等
- 状态：待处理
- 依赖：无

**任务2：设计接口规范**
- 标题：设计接口规范
- 描述：设计后端API的接口规范，包括请求格式、响应格式、错误处理
- 状态：待处理
- 依赖：任务1（需要知道API的限制才能设计接口）

**任务3：实现后端接口**
- 标题：实现后端接口
- 描述：根据设计的规范实现后端API，调用天气API获取数据
- 状态：待处理
- 依赖：任务2（需要先有接口规范）

**任务4：实现前端界面**
- 标题：实现前端界面

## 7. 执行过程中的关键决策点

### 7.1 是否需要创建任务列表？

**需要创建的情况**
- 任务涉及3个或更多独立步骤
- 步骤之间有明确的依赖关系
- 需要追踪整体进度
- 可能有并行执行的机会

**不需要创建的情况**
- 单一简单的任务
- 快速的一次性操作
- 任务之间没有依赖关系

### 7.2 如何拆分任务？

**拆分原则**
1. **独立性**：每个任务应该有明确的边界
2. **可验证**：任务完成与否应该可以明确判断
3. **适当粒度**：不要太小（管理成本高）也不要太大（难以追踪）
4. **依赖最小化**：尽量减少任务间的依赖关系

**常见拆分方式**
- 按功能模块拆分（前端、后端、数据库）
- 按开发阶段拆分（调研、设计、实现、测试）
- 按技术领域拆分（UI、API、配置）

### 7.3 如何设置依赖关系？

**设置原则**
1. **必要性**：只有真正必须的依赖才设置
2. **避免循环**：确保依赖图是无环的
3. **最小化**：使用最少的依赖满足需求

**常见依赖模式**
- 顺序依赖：A → B → C
- 分支依赖：A → B, A → C
- 汇合依赖：A → C, B → C

### 7.4 如何选择执行顺序？

**优先级判断**
1. 优先执行阻塞其他任务最多的任务
2. 优先执行耗时较长的任务（可以后台运行）
3. 优先执行风险较高的任务（早发现问题）

**并行策略**
1. 识别依赖图中同一层级的任务
2. 同时启动多个后台子代理
3. 定期检查完成情况
4. 处理先完成的任务，释放后续任务

### 7.5 如何处理任务失败？

**失败检测**
- 子代理返回错误状态
- 子代理超时未响应
- 子代理抛出异常

**处理策略**
1. 记录失败原因和上下文
2. 将任务状态回退到"待处理"
3. 清空任务所有者
4. 在元数据中记录失败信息
5. 考虑是否需要调整任务描述或拆分任务

## 8. 常见场景处理

### 8.1 场景：发现遗漏的任务

**情况描述**
执行过程中发现需要增加一个新任务，比如"配置环境变量"。

**处理步骤**
1. 调用 TaskCreate 创建新任务
2. 分析新任务的依赖关系
3. 如果有依赖，调用 TaskUpdate 设置 addBlockedBy
4. 更新整体计划

**示例**

发现后端实现需要先配置环境变量：
1. 创建任务"配置环境变量"
2. 设置"后端实现"依赖"配置环境变量"
3. 由于"后端实现"已在进行中，可能需要暂停或标记新任务为高优先级

### 8.2 场景：任务比预期复杂

**情况描述**
执行任务时发现需要的工作量远超预期。

**处理步骤**
1. 将当前任务状态回退到"待处理"
2. 将任务拆分成多个更小的子任务
3. 设置子任务之间的依赖关系
4. 按新计划执行

**示例**

"实现后端接口"太复杂，拆分为：
- 任务3a：创建基础路由结构
- 任务3b：实现 API 调用逻辑
- 任务3c：实现数据转换和格式化
- 任务3d：实现错误处理

### 8.3 场景：依赖的任务失败

**情况描述**
任务A失败了，但任务B依赖任务A。

**处理步骤**
1. 检查任务A失败的原因
2. 尝试修复任务A并重新执行
3. 如果无法修复，考虑：
   - 调整任务B的依赖（如果可能）
   - 创建替代方案的任务
   - 向用户报告阻塞情况

### 8.4 场景：多个子代理同时工作

**情况描述**
启动了多个后台子代理并行执行任务。

**监控要点**
1. 定期检查每个子代理的进度
2. 处理先完成的子代理
3. 更新对应的任务状态
4. 检查是否有新任务解锁
5. 处理失败的子代理

**注意事项**
- 避免同时启动过多子代理（资源限制）
- 子代理之间可能需要协调（避免同时修改同一文件）
- 完成顺序可能影响后续计划

### 8.5 场景：需要用户输入

**情况描述**
子代理执行过程中需要用户提供信息或做出决定。

**处理方式**

如果子代理支持交互：
1. 子代理向用户提问
2. 等待用户回答
3. 继续执行

如果子代理不支持交互（后台模式）
1. 子代理标记需要输入
2. 主代理检查子代理状态
3. 主代理向用户提问
4. 恢复子代理并传入用户回答

### 8.6 场景：需要取消任务

**情况描述**
用户或系统决定取消某个任务。

**处理步骤**

1. 调用 TaskUpdate 设置状态为 `cancelled`
2. 如果任务正在进行中：
   - 子代理会被自动取消
   - 清理代理资源
3. 通知依赖此任务的任务（它们可能需要重新规划）
4. 记录取消原因

**示例**

```typescript
// 取消一个正在进行中的任务
await TaskUpdate({
  taskId: '3',
  status: 'cancelled',
  reason: '需求变更，此功能不再需要',
  updatedBy: 'user',
});

// 系统会自动：
// 1. 取消关联的子代理
// 2. 通知依赖任务3的任务
// 3. 记录历史
```

### 8.7 场景：任务需要重试

**情况描述**
任务执行失败，但根据配置可以自动重试。

**处理步骤**

1. 系统自动检查重试配置
2. 如果可以重试：
   - 计算重试延迟（指数退避）
   - 清空 owner，重置状态为 pending
   - 延迟后重新执行
3. 如果达到最大重试次数：
   - 标记为 `failed`
   - 通知依赖任务

**示例**

```typescript
// 创建带重试配置的任务
await TaskCreate({
  subject: '调用外部API',
  description: '调用第三方API获取数据',
  retryConfig: {
    maxRetries: 3,
    retryDelayMs: 5000,
    backoffMultiplier: 2,
    retryOn: ['timeout', 'network_error'],
  },
});

// 第一次失败后，系统会：
// 1. 等待 5 秒后重试
// 第二次失败后，系统会：
// 1. 等待 10 秒后重试 (5 * 2)
// 第三次失败后，系统会：
// 1. 标记任务为 failed
```

### 8.8 场景:紧急任务处理

**情况描述**
有一个紧急任务需要立即处理，即使有其他任务正在进行。

**处理步骤**

1. 创建任务时设置优先级为 `critical`
2. 紧急任务可以忽略非完成的依赖（会有警告）
3. 优先执行紧急任务

**示例**

```typescript
// 创建紧急任务
await TaskCreate({
  subject: '修复生产环境崩溃',
  description: '紧急修复导致服务不可用的问题',
  priority: 'critical',
});

// 即使有依赖未完成，紧急任务也可以开始
// 但系统会发出警告
const taskDetail = await TaskGet('emergency-task-id');
console.log(taskDetail.canStart);
// { canStart: true, reason: '警告: 有 2 个依赖未完成' }
```

### 8.9 场景:更新任务进度

**情况描述**
长时间运行的任务需要报告当前进度。

**处理步骤**

1. 子代理在执行过程中调用进度更新接口
2. 主代理显示进度给用户
3. 宣布检查点完成

4. 完成时自动设置进度为 100%

**示例**

```typescript
// 开始执行任务
await TaskUpdate({
  taskId: '1',
  status: 'in_progress',
  owner: 'agent-001',
});

// 更新进度
await updateTaskProgress('1', 30);  // 30%
await updateTaskProgress('1', 50, '数据处理完成');  // 50% + 完成检查点

// 完成检查点
await completeCheckpoint('1', 'checkpoint-1');

// 任务完成时自动设置 100%
await TaskUpdate({
  taskId: '1',
  status: 'completed',
});
```

### 8.10 场景:查看任务历史

**情况描述**
需要查看任务的所有变更历史，用于问题排查。

**处理步骤**

1. 使用 TaskGet 获取任务详情（包含历史）
2. 分析状态变更时间线
3. 查看失败原因和重试记录

  }

### 8.11 场景:按标签筛选任务

**情况描述**
需要查看特定模块或类型的所有任务。

**处理步骤**

1. 按标签名筛选

3. 组合多个条件筛选

### 8.12 场景:处理任务超时

**情况描述**
任务执行时间超过预期，需要处理。

**处理步骤**

1. 系统检测到超时
2. 根据任务配置决定处理方式：
   - 重试（如果配置了重试）
   - 失败（如果不需要重试)
   - 延长超时（如果任务支持）
3. 记录超时事件

**示例**

```typescript
// 子代理在执行循环中检测超时
async function executeSync(runtime, context, request) {

    // 检查超时

      runtime.status = 'timeout';
      return {
        agentId: runtime.agentId,
        status: 'timeout',
        message: `任务执行超时 (${timeout}ms)`,
      };
    }

    // ... 继续执行 ...
  }
}
```

## 9. 问题诊断与排查## 9. 问题诊断与排查

### 9.1 任务一直处于"待处理"状态

**可能原因**
1. 任务被其他任务阻塞
2. 所有代理都在忙碌
3. 任务描述不清晰，代理无法理解

**排查步骤**
1. 调用 TaskGet 检查任务的阻塞情况
2. 检查阻塞任务的状态
3. 如果阻塞任务已完成但依赖关系未更新，手动更新
4. 检查任务描述是否清晰

### 9.2 子代理执行超时

**可能原因**
1. 任务过于复杂
2. 代理陷入循环
3. 网络请求超时

**排查步骤**
1. 检查子代理的输出日志
2. 分析代理的执行轨迹
3. 考虑简化任务或增加超时时间
4. 考虑拆分任务

### 9.3 任务状态不一致

**可能原因**
1. 并发修改冲突
2. 子代理完成但未正确更新状态
3. 系统异常中断

**排查步骤**
1. 调用 TaskList 检查整体状态
2. 比较任务状态和实际完成情况
3. 手动修正不一致的状态
4. 检查是否有多个代理同时操作同一任务

### 9.4 依赖关系出现循环

**可能原因**
1. 设置依赖时未考虑间接影响
2. 多次修改依赖关系导致

**排查步骤**
1. 系统在添加依赖时会自动检测循环
2. 如果已经存在循环（历史数据问题），需要手动解除
3. 分析依赖图，找到可以移除的依赖边
4. 重新设置正确的依赖关系

### 9.5 子代理结果不符合预期

**可能原因**
1. 任务描述不清晰
2. 代理理解偏差
3. 缺少必要的上下文

**排查步骤**
1. 检查子代理的执行过程
2. 分析代理的决策点
3. 改进任务描述，增加具体要求
4. 提供更多上下文信息
5. 考虑使用不同类型的代理

### 10.1 核心要点回顾

**两套系统的定位**
- TaskCreate/Update/Get/List：管理"做什么"，负责任务规划和追踪
- Task：执行"怎么做"，负责实际工作

**关键流程**
1. 创建任务列表，设置依赖关系
2. 查找可执行的任务
3. 认领任务
4. 选择合适的子代理执行
5. 更新任务状态
6. 重复直到完成

**核心判断**
- 任务是否可以开始：检查状态、依赖、归属
- 子代理是否完成：检查输出、工具调用、明确标记
- 是否可以并行：分析依赖图，找出独立任务

### 10.2 最佳实践

1. **合理拆分任务**：粒度适中，边界清晰
2. **正确设置依赖**：最小化依赖，避免循环
3. **选择合适的代理**：根据任务特征选择
4. **善用并行执行**：识别独立任务，提高效率
5. **及时更新状态**：保持任务列表的准确性
6. **监控执行过程**：定期检查进度，及时处理问题
